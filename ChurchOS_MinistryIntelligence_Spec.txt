0.  Context — What We Are Building & Why
Church OS currently runs two operational layers: the Member App (individual devotions and SOAP journaling) and Mission Control (pastoral oversight for admins). The complete database foundation was established on March 11, 2026 — 16 core tables, 4 automatic triggers, and role-consistency enforcement are all in place and verified.
 
What is missing is the middle layer: the Ministry Intelligence Layer. Today, if leadership wants to know how Children's Ministry attendance is trending, or how many people the Evangelism team reached last month, someone has to manually enter that data into the admin dashboard. This makes the admin the bottleneck for every operational insight, and it means ministry leaders have no ownership over their own performance data.
 
This document specifies every database table, trigger, function, role rule, and frontend pattern required to build the distributed ministry reporting infrastructure. After this is implemented, Church OS functions as a genuine Ministry Intelligence Platform — where data is captured at the source by the people closest to each ministry, flows automatically into Mission Control, and gives leadership real-time analytics without any manual entry.

1.  The Three-Layer Architecture
The system operates across three clearly separated layers. Each has a distinct user base, a distinct data responsibility, and a distinct interface. Data flows strictly upward — from members to ministry leaders to Mission Control. Nothing requires admin manual entry.

3.1  ministries — Master Ministry Registry
This is the most foundational new table. Currently ministry names exist only as string constants in constants.ts with no database representation. This table gives each ministry a proper identity: its own UUID, its own leader, its own color and icon for theming, and its own slug for URL routing. Every other new table references this table by ID.
 
CREATE TABLE IF NOT EXISTS ministries (
  id          UUID        PRIMARY KEY DEFAULT uuid_generate_v4(),
  org_id      UUID        NOT NULL REFERENCES organizations(id) ON DELETE CASCADE,
  name        TEXT        NOT NULL,
  slug        TEXT        NOT NULL,  -- URL-safe identifier: 'worship', 'childrens', 'ushers'
  description TEXT,
  category    TEXT        DEFAULT 'general',
  -- category options: worship | care | outreach | admin | media
  leader_id   UUID        REFERENCES auth.users(id),
  color       TEXT        DEFAULT '#6366F1',  -- Hex color for dashboard theming
  icon        TEXT        DEFAULT 'users',    -- Lucide icon name
  is_active   BOOLEAN     DEFAULT TRUE,
  created_at  TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE(org_id, slug)
);

3.2  ministry_reports — Central Operational Report Store
This is the most important table in the entire Ministry Intelligence Layer. Every form submission from every ministry leader across all 12 ministries lands here as a single row. The report_type column identifies which form was submitted. The data JSONB column stores all form field values. This design means you never need to create a separate table for each new form — you simply add a new report_type value and a matching form_template row.
 
CREATE TABLE IF NOT EXISTS ministry_reports (
  id           UUID        PRIMARY KEY DEFAULT uuid_generate_v4(),
  org_id       UUID        NOT NULL REFERENCES organizations(id) ON DELETE CASCADE,
  ministry_id  UUID        NOT NULL REFERENCES ministries(id) ON DELETE CASCADE,
  submitted_by UUID        NOT NULL REFERENCES auth.users(id),
  report_type  TEXT        NOT NULL,
  -- Valid report_type values:
  -- 'attendance'           General service attendance
  -- 'activity'             Weekly ministry activity summary
  -- 'event'                Post-event report
  -- 'equipment'            Resource/equipment status
  -- 'outreach'             Evangelism outreach report
  -- 'children_class'       Sunday school class report
  -- 'children_safety'      Child safety incident report
  -- 'media_stream'         Livestream analytics
  -- 'offering_recon'       Usher offering reconciliation
  -- 'counseling'           Pastoral care session log
  -- 'prayer_meeting'       Prayer ministry attendance
  service_date DATE        NOT NULL DEFAULT CURRENT_DATE,
  data         JSONB       NOT NULL DEFAULT '{}',
  -- Example data for 'attendance' report_type:
  -- { "total_adults": 47, "total_children": 12,
  --   "first_time_visitors": 3, "notes": "Good service" }
  status       TEXT        DEFAULT 'submitted',
  -- status options: submitted | reviewed | flagged
  reviewed_by  UUID        REFERENCES auth.users(id),
  reviewed_at  TIMESTAMPTZ,
  created_at   TIMESTAMPTZ DEFAULT NOW()
);

-- Performance indexes
CREATE INDEX IF NOT EXISTS idx_ministry_reports_ministry_date
  ON ministry_reports(ministry_id, service_date DESC);
CREATE INDEX IF NOT EXISTS idx_ministry_reports_org_type
  ON ministry_reports(org_id, report_type, service_date DESC);
CREATE INDEX IF NOT EXISTS idx_ministry_reports_data
  ON ministry_reports USING GIN(data);  -- enables fast JSONB field queries

3.4  ministry_announcements — Two-Way Communication Channel
This table enables structured communication between Mission Control and individual ministries, replacing informal WhatsApp messages with a traceable in-system channel. Admins push announcements downward to specific ministries or to all ministries at once. Ministry leaders post updates upward to Mission Control. Every message has a direction, priority, and read status.
 
CREATE TABLE IF NOT EXISTS ministry_announcements (
  id          UUID    PRIMARY KEY DEFAULT uuid_generate_v4(),
  org_id      UUID    NOT NULL REFERENCES organizations(id) ON DELETE CASCADE,
  ministry_id UUID    REFERENCES ministries(id),
  -- NULL ministry_id = broadcast to ALL ministries
  author_id   UUID    NOT NULL REFERENCES auth.users(id),
  direction   TEXT    NOT NULL DEFAULT 'downward',
  -- direction: 'downward' = Admin to Ministry
  --            'upward'   = Ministry to Admin
  --            'lateral'  = Ministry to Ministry (Phase 2)
  title       TEXT    NOT NULL,
  body        TEXT    NOT NULL,
  priority    TEXT    DEFAULT 'normal',
  -- priority: normal | urgent | critical
  is_read     BOOLEAN DEFAULT FALSE,
  created_at  TIMESTAMPTZ DEFAULT NOW()
);
 
3.5  form_templates — Dynamic Form Definition Engine
This table stores the structure of each form as a JSONB fields array. It is what makes the entire system scalable: when leadership needs a new form type, an admin defines it here without any developer involvement. The DynamicFormRenderer component on the frontend reads a form_templates row and renders the correct inputs automatically. When a ministry leader submits the form, the component writes to ministry_reports using the report_type from this table.
 
CREATE TABLE IF NOT EXISTS form_templates (
  id          UUID    PRIMARY KEY DEFAULT uuid_generate_v4(),
  org_id      UUID    NOT NULL REFERENCES organizations(id) ON DELETE CASCADE,
  ministry_id UUID    REFERENCES ministries(id),
  -- NULL = universal form available to all ministries
  name        TEXT    NOT NULL,
  report_type TEXT    NOT NULL,  -- must match a valid ministry_reports.report_type value
  description TEXT,
  fields      JSONB   NOT NULL DEFAULT '[]',
  -- fields is an ordered array of field definition objects:
  -- [
  --   {
  --     "name":        "total_adults",
  --     "label":       "Total Adults Present",
  --     "type":        "number",
  --     "required":    true,
  --     "placeholder": "Enter headcount"
  --   },
  --   {
  --     "name":    "service_type",
  --     "label":   "Service Type",
  --     "type":    "select",
  --     "options": ["Sunday Service","Midweek","Special Event","Training"],
  --     "required": true
  --   }
  -- ]
  is_active   BOOLEAN DEFAULT TRUE,
  version     INT     DEFAULT 1,
  created_at  TIMESTAMPTZ DEFAULT NOW()
);
 
3.6  ministry_analytics — Pre-Computed Analytics Cache
Rather than running expensive live aggregate queries against ministry_reports every time Mission Control loads, this table stores pre-computed weekly and monthly analytics per ministry. A trigger fires on every new report submission to update this cache automatically. Mission Control reads from here for near-instant dashboard loads.
 
CREATE TABLE IF NOT EXISTS ministry_analytics (
  id              UUID    PRIMARY KEY DEFAULT uuid_generate_v4(),
  org_id          UUID    NOT NULL REFERENCES organizations(id) ON DELETE CASCADE,
  ministry_id     UUID    NOT NULL REFERENCES ministries(id) ON DELETE CASCADE,
  period_start    DATE    NOT NULL,
  period_end      DATE    NOT NULL,
  period_type     TEXT    DEFAULT 'weekly',  -- weekly | monthly | quarterly
  avg_attendance  NUMERIC(8,2) DEFAULT 0,
  total_reports   INT     DEFAULT 0,
  volunteer_count INT     DEFAULT 0,
  event_count     INT     DEFAULT 0,
  salvations      INT     DEFAULT 0,
  visitor_count   INT     DEFAULT 0,
  health_score    INT     DEFAULT 0,  -- 0-100 composite score
  computed_at     TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE(ministry_id, period_start, period_type)
);

4.1  Auto-update analytics cache on every report submission
Every time a ministry leader submits any form, this trigger fires and upserts the analytics cache for that ministry's current week. Mission Control's Ministry Hub page will always show current-week data without any manual refresh or scheduled job.
 
CREATE OR REPLACE FUNCTION update_ministry_analytics_on_report()
RETURNS TRIGGER AS $$
DECLARE
  v_period_start  DATE    := date_trunc('week', NEW.service_date)::DATE;
  v_period_end    DATE    := (date_trunc('week', NEW.service_date) + INTERVAL '6 days')::DATE;
  v_attendance    INT     := COALESCE((NEW.data->>'total_adults')::INT, 0)
                           + COALESCE((NEW.data->>'total_children')::INT, 0);
  v_visitors      INT     := COALESCE((NEW.data->>'first_time_visitors')::INT, 0);
  v_salvations    INT     := COALESCE((NEW.data->>'salvations')::INT, 0);
BEGIN
  INSERT INTO ministry_analytics (
    org_id, ministry_id, period_start, period_end,
    period_type, avg_attendance, total_reports, visitor_count, salvations
  ) VALUES (
    NEW.org_id, NEW.ministry_id, v_period_start, v_period_end,
    'weekly', v_attendance, 1, v_visitors, v_salvations
  )
  ON CONFLICT (ministry_id, period_start, period_type) DO UPDATE SET
    total_reports  = ministry_analytics.total_reports + 1,
    avg_attendance = ROUND(
      (ministry_analytics.avg_attendance * ministry_analytics.total_reports
       + v_attendance) / (ministry_analytics.total_reports + 1), 2),
    visitor_count  = ministry_analytics.visitor_count  + v_visitors,
    salvations     = ministry_analytics.salvations     + v_salvations,
    computed_at    = NOW();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

DROP TRIGGER IF EXISTS ministry_report_analytics_trigger ON ministry_reports;
CREATE TRIGGER ministry_report_analytics_trigger
AFTER INSERT ON ministry_reports
FOR EACH ROW EXECUTE FUNCTION update_ministry_analytics_on_report();
 
4.2  Alert Mission Control when a report is flagged
If a ministry leader marks any submitted report as 'flagged' — indicating a safety incident, critical resource failure, or urgent pastoral matter — this trigger immediately creates a member_alert row so Mission Control shows the escalation in the Care Alerts panel without requiring any manual notification.
 
CREATE OR REPLACE FUNCTION alert_admin_on_flagged_report()
RETURNS TRIGGER AS $$
DECLARE
  v_ministry_name TEXT;
BEGIN
  IF NEW.status = 'flagged'
     AND (OLD IS NULL OR OLD.status IS DISTINCT FROM 'flagged') THEN
    SELECT name INTO v_ministry_name
    FROM ministries WHERE id = NEW.ministry_id;
    INSERT INTO member_alerts (member_id, org_id, alert_type, severity, message)
    VALUES (
      NEW.submitted_by,
      NEW.org_id,
      'Ministry Report Flag',
      'critical',
      v_ministry_name || ' submitted a flagged report — '
        || NEW.report_type || ' on '
        || TO_CHAR(NEW.service_date, 'Mon DD, YYYY')
    );
  END IF;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

DROP TRIGGER IF EXISTS flagged_report_alert_trigger ON ministry_reports;
CREATE TRIGGER flagged_report_alert_trigger
AFTER INSERT OR UPDATE OF status ON ministry_reports
FOR EACH ROW EXECUTE FUNCTION alert_admin_on_flagged_report();
 
4.3  Compute ministry health score (called by intelligence sweep)
This function is invoked by the AI Command Center's Run Sweep button via the intelligence engine in route.ts. It computes a 0-100 health score for each ministry based on reporting consistency (40%), attendance trend (35%), and active volunteer count (25%). The score is written back into ministry_analytics and surfaced on the Mission Control Ministry Hub.
 
CREATE OR REPLACE FUNCTION compute_ministry_health_score(p_ministry_id UUID)
RETURNS INT AS $$
DECLARE
  v_reports_30d   INT;
  v_avg_attend    NUMERIC;
  v_volunteers    INT;
  v_consistency   INT;
  v_attend_score  INT;
  v_engage_score  INT;
  v_final         INT;
BEGIN
  SELECT COUNT(*) INTO v_reports_30d
  FROM ministry_reports
  WHERE ministry_id = p_ministry_id
    AND service_date >= CURRENT_DATE - INTERVAL '30 days';

  SELECT COALESCE(AVG(
    COALESCE((data->>''total_adults'')::INT, 0)
    + COALESCE((data->>''total_children'')::INT, 0)
  ), 0) INTO v_avg_attend
  FROM ministry_reports
  WHERE ministry_id = p_ministry_id
    AND service_date >= CURRENT_DATE - INTERVAL '30 days';

  SELECT COUNT(*) INTO v_volunteers
  FROM ministry_members
  WHERE ministry_id = p_ministry_id AND is_active = TRUE;

  v_consistency  := LEAST(100, v_reports_30d  * 25);  -- 4 reports/mo = 100
  v_attend_score := LEAST(100, v_avg_attend::INT * 2); -- 50 avg = 100
  v_engage_score := LEAST(100, v_volunteers    * 10);  -- 10 volunteers = 100

  v_final := ROUND(
    (v_consistency  * 0.40) +
    (v_attend_score * 0.35) +
    (v_engage_score * 0.25)
  );

  UPDATE ministry_analytics
  SET health_score = v_final, computed_at = NOW()
  WHERE ministry_id    = p_ministry_id
    AND period_type    = 'monthly'
    AND period_start   = date_trunc('month', CURRENT_DATE)::DATE;

  RETURN v_final;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

5.  Row Level Security Policies
Enable RLS on all new tables and add the policies below. These follow the same pattern established in the March 11 session. Regular users are restricted to their own data by RLS. All admin dashboard pages use supabaseAdmin (service role key) which bypasses RLS entirely.
 
-- Enable RLS
ALTER TABLE ministries              ENABLE ROW LEVEL SECURITY;
ALTER TABLE ministry_reports        ENABLE ROW LEVEL SECURITY;
ALTER TABLE ministry_events         ENABLE ROW LEVEL SECURITY;
ALTER TABLE ministry_announcements  ENABLE ROW LEVEL SECURITY;
ALTER TABLE form_templates          ENABLE ROW LEVEL SECURITY;
ALTER TABLE ministry_analytics      ENABLE ROW LEVEL SECURITY;

-- ministries: any authenticated user can read the ministry list
DO $$ BEGIN
IF NOT EXISTS (SELECT 1 FROM pg_policies WHERE policyname = 'Read ministries') THEN
  CREATE POLICY "Read ministries" ON ministries
  FOR SELECT USING (auth.uid() IS NOT NULL);
END IF; END $$;

-- ministry_reports: leaders/assistants see their own ministry's reports only
DO $$ BEGIN
IF NOT EXISTS (SELECT 1 FROM pg_policies WHERE policyname = 'Leaders access own ministry reports') THEN
  CREATE POLICY "Leaders access own ministry reports" ON ministry_reports
  FOR ALL USING (
    auth.uid() = submitted_by
    OR EXISTS (
      SELECT 1 FROM ministry_members mm
      WHERE mm.user_id = auth.uid()
        AND mm.ministry_id = ministry_reports.ministry_id
        AND mm.ministry_role IN ('leader','assistant')
        AND mm.is_active = TRUE
    )
  );
END IF; END $$;

-- form_templates: any authenticated user can read active templates
DO $$ BEGIN
IF NOT EXISTS (SELECT 1 FROM pg_policies WHERE policyname = 'Read active form templates') THEN
  CREATE POLICY "Read active form templates" ON form_templates
  FOR SELECT USING (auth.uid() IS NOT NULL AND is_active = TRUE);
END IF; END $$;

-- ministry_announcements: members see announcements for their ministries
DO $$ BEGIN
IF NOT EXISTS (SELECT 1 FROM pg_policies WHERE policyname = 'Members read ministry announcements') THEN
  CREATE POLICY "Members read ministry announcements" ON ministry_announcements
  FOR SELECT USING (
    ministry_id IS NULL  -- broadcast to all
    OR EXISTS (
      SELECT 1 FROM ministry_members mm
      WHERE mm.user_id = auth.uid()
        AND mm.ministry_id = ministry_announcements.ministry_id
    )
  );
END IF; END $$;

-- ministry_analytics: any authenticated user can read cached analytics
DO $$ BEGIN
IF NOT EXISTS (SELECT 1 FROM pg_policies WHERE policyname = 'Read ministry analytics') THEN
  CREATE POLICY "Read ministry analytics" ON ministry_analytics
  FOR SELECT USING (auth.uid() IS NOT NULL);
END IF; END $$;

7.2  New File Required: src/lib/ministry-auth.ts
Create this file as the parallel of the existing src/lib/admin-auth.ts. It guards all /ministry-dashboard/[slug]/* routes. The logic is: extract the ministry slug from the URL, find the logged-in user's record in ministry_members for that ministry, and return their role. If no record exists, redirect to /ministry-dashboard with an 'access denied' message.
 
-- Query that MinistryAuth runs to validate dashboard access
-- Parameters: slug from URL params, user UUID from auth session
SELECT
  mm.ministry_role,
  mm.ministry_id,
  m.name        AS ministry_name,
  m.slug,
  m.color,
  m.icon,
  m.description
FROM ministry_members mm
JOIN ministries m ON m.id = mm.ministry_id
WHERE mm.user_id   = :user_id    -- from auth.uid()
  AND m.slug       = :slug       -- from URL params
  AND mm.is_active = TRUE
  AND m.is_active  = TRUE
LIMIT 1;

-- Result interpretation:
-- No row returned          → redirect to /ministry-dashboard (not a member)
-- ministry_role = 'member' → read-only view, no form submission
-- ministry_role = 'volunteer'  → can submit forms only
-- ministry_role = 'assistant'  → can submit forms + view team
-- ministry_role = 'leader'     → full dashboard access

8.1  Ministry Hub — Live Ministry Health Dashboard
The Ministry Hub page must query the ministries and ministry_analytics tables to show a health card per ministry with its score, recent activity status, and reporting compliance.
 
-- Ministry Hub overview — one card per ministry
SELECT
  m.id, m.name, m.slug, m.color, m.icon,
  p.name                                      AS leader_name,
  COUNT(DISTINCT mm.id)                       AS volunteer_count,
  ma.health_score,
  ma.avg_attendance,
  ma.total_reports,
  ma.salvations,
  -- Date of most recent report submission
  (SELECT MAX(mr.service_date)
   FROM ministry_reports mr
   WHERE mr.ministry_id = m.id)               AS last_report_date,
  -- Flag if no report submitted in 14 days
  CASE WHEN (SELECT MAX(mr.service_date)
             FROM ministry_reports mr
             WHERE mr.ministry_id = m.id)
            < CURRENT_DATE - INTERVAL '14 days'
    THEN TRUE ELSE FALSE
  END                                          AS reporting_overdue
FROM ministries m
LEFT JOIN profiles p         ON p.id = m.leader_id
LEFT JOIN ministry_members mm ON mm.ministry_id = m.id AND mm.is_active = TRUE
LEFT JOIN ministry_analytics ma ON ma.ministry_id = m.id
  AND ma.period_type  = 'monthly'
  AND ma.period_start = date_trunc('month', CURRENT_DATE)::DATE
WHERE m.org_id = :org_id AND m.is_active = TRUE
GROUP BY m.id, p.name, ma.health_score, ma.avg_attendance,
         ma.total_reports, ma.salvations
ORDER BY m.name;
 
8.2  Growth Intelligence — Real Salvations from Evangelism Reports
The Growth Intelligence funnel currently shows zero because no outreach data has been entered. Now that the Evangelism Ministry will submit outreach reports via their dashboard, this query reads from ministry_reports to populate the funnel with real data.
 
-- Salvation & outreach trend (past 6 months)
SELECT
  DATE_TRUNC('month', service_date)         AS month,
  SUM((data->>'salvations')::INT)           AS total_salvations,
  SUM((data->>'people_reached')::INT)       AS people_reached,
  SUM((data->>'contacts_collected')::INT)   AS contacts,
  COUNT(*)                                   AS outreach_events
FROM ministry_reports
WHERE org_id      = :org_id
  AND report_type = 'outreach'
  AND service_date >= CURRENT_DATE - INTERVAL '6 months'
GROUP BY DATE_TRUNC('month', service_date)
ORDER BY month DESC;
 
8.3  Attendance Reconciliation — Usher Headcount vs Digital Check-In
The existing vw_attendance_reconciliation view tried to compare usher headcounts against digital attendance but failed because ushers never submitted digitally. Now that ushers will submit Service Headcount Reports through their dashboard, replace the view with this query.
 
-- Attendance reconciliation: digital vs usher headcount
SELECT
  ar.event_date,
  ar.digital_count,
  COALESCE((mr.data->>'total_adults')::INT, 0)
  + COALESCE((mr.data->>'total_children')::INT, 0)   AS usher_headcount,
  ar.digital_count
  - COALESCE((mr.data->>'total_adults')::INT, 0)
  - COALESCE((mr.data->>'total_children')::INT, 0)   AS unregistered_gap
FROM (
  SELECT event_date, COUNT(*) AS digital_count
  FROM attendance_records
  WHERE org_id = :org_id
  GROUP BY event_date
) ar
LEFT JOIN ministry_reports mr
  ON  mr.service_date = ar.event_date
  AND mr.report_type  = 'offering_recon'
  AND mr.org_id       = :org_id
  AND mr.ministry_id  = (
    SELECT id FROM ministries
    WHERE slug = 'ushers' AND org_id = :org_id
  )
ORDER BY ar.event_date DESC
LIMIT 12;

11.  Critical Engineering Rules
These rules apply to all work on the Ministry Intelligence Layer. They extend the Golden Rules established in the March 11 session report.
 
	•	All /ministry-dashboard/* pages must use the regular supabase client (not supabaseAdmin) so that RLS correctly restricts what ministry leaders can see. Only Mission Control admin pages use supabaseAdmin.
	•	Never create a separate database table for each ministry's data. All reports go into ministry_reports. The report_type column + data JSONB is the correct pattern.
	•	Never hardcode ministry slugs as string literals anywhere in component files. Always reference the MINISTRIES constant from constants.ts. This ensures a single source of truth.
	•	The children_table form field type stores an array of objects in the JSONB data column. Make sure the DynamicFormRenderer serializes this correctly before submitting — do not try to flatten the array into individual fields.
	•	The ministry_analytics table is a cache, not a source of truth. If analytics look wrong, check whether the trigger fired correctly on the last ministry_reports INSERT. Never manually update ministry_analytics rows — only the trigger should write to them.
	•	Every new form type added to form_templates must also have its report_type string added to the ministry_reports column comment so the team knows all valid values.
